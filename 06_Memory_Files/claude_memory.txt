**Problem Solved**: LLM→TTS handoff was broken in modular mode

- ASR transcribed speech (Deepgram working)
- LLM generated responses (Cerebras/Groq working)
- **BUT**: Responses never reached TTS
- Errors: "Unknown data type: undefined" OR "Error parsing LLM service response"

**Root Cause**: AVR Core expects specific JSON format for buffered LLM responses

### The Critical Fix: Response Format

**WORKING FORMAT** (DO NOT CHANGE):
```javascript
res.json({ type: 'text', content: responseText })
```

**Failed Formats**:
- `{ response: text }` ❌ Unknown data type error
- `{ content: text }` ❌ Unknown data type error  
- SSE: `data: {"content": "..."}\n\n` ❌ JSON parse error
- SSE: `data: {"type":"text","content":"..."}\n\n` ❌ JSON parse error

**Why It Works**:
- AVR Core in buffered mode expects plain parseable JSON (not SSE)
- Requires `type: 'text'` field to identify content type
- Requires `content` field with actual LLM response text

### Groq LLM Provider Implementation

**File**: `/home/user/Desktop/Projects_Organized/02_AVR_Voice_Platform/avr-app/custom-providers/avr-llm-groq/index.js`

**Critical Code** (EXACT WORKING VERSION):
```javascript
const express = require('express');
const { OpenAI } = require('openai');
const app = express();
app.use(express.json());

const GROQ_API_KEY = process.env.GROQ_API_KEY;
const MODEL = process.env.GROQ_MODEL || 'llama3-8b-8192';
const TEMPERATURE = parseFloat(process.env.TEMPERATURE || '0.7');

const openai = new OpenAI({
  apiKey: GROQ_API_KEY,
  baseURL: 'https://api.groq.com/openai/v1',
});

app.get('/health', (req, res) => {
  res.json({ status: 'ok', provider: 'groq', model: MODEL });
});

app.post('/prompt-stream', async (req, res) => {
  try {
    const { messages = [], systemPrompt = 'You are a helpful assistant.' } = req.body;
    const formattedMessages = [
      { role: 'system', content: systemPrompt },
      ...messages
    ];

    console.log(`[Groq] /prompt-stream with ${messages.length} messages`);

    const completion = await openai.chat.completions.create({
      model: MODEL,
      messages: formattedMessages,
      temperature: TEMPERATURE,
      stream: false
    });

    const responseText = completion.choices[0]?.message?.content || '';
    console.log(`[Groq] Response: ${responseText.substring(0, 50)}...`);

    // CRITICAL: AVR expects this exact format
    res.json({ type: 'text', content: responseText });
  } catch (error) {
    console.error('[Groq] Error:', error.message);
    res.status(500).json({ error: error.message });
  }
});

app.listen(6002, () => {
  console.log('[Groq LLM Provider] Running on port 6002');
});
```

## Phase 16: AudioSocket Recording Fix - SOLVED (2025-11-13)
================================================================================

### Critical Discovery: Recording Breakthrough

**Problem**: All recordings were 44 bytes (empty WAV headers)
- Neither AVR nor Pipecat bots have built-in audio WAV recording
- Both systems only save transcripts (text conversation logs)
- Session summary incorrectly suggested Pipecat had "built-in recording"

**Root Cause**: AudioSocket Bypass
- AudioSocket creates bidirectional stream EXTERNAL to Asterisk's channel framework
- MixMonitor, Monitor, and similar apps can't see audio flowing through AudioSocket
- The `,b` option (record only TO bot) especially problematic with bidirectional external streams

### The Fix: AUDIOHOOK_INHERIT

**Solution**: Add `Set(AUDIOHOOK_INHERIT(MixMonitor)=yes)` before MixMonitor in dialplan

**What it does**: Makes recording hook "follow" the channel when bridged to external applications

**Implementation** (applied to all 20 bots: 9092-9111):

```diff
exten => 909X,1,NoOp(Bridging to bot on port 909X)
 same => n,Answer()
 same => n,Set(AUDIOSOCKET_UUID=...)
 same => n,Set(CALL_ID=...)
 same => n,Set(RECORDING_FILE=...)
+ same => n,Set(AUDIOHOOK_INHERIT(MixMonitor)=yes)
- same => n,MixMonitor(${RECORDING_FILE},b)
+ same => n,MixMonitor(${RECORDING_FILE})
 same => n,AudioSocket(${AUDIOSOCKET_UUID},172.17.0.1:909X)
 same => n,StopMixMonitor()
 same => n,Hangup()
```

### Key Learnings

1. **AudioSocket is inherently difficult to record**
   - External to Asterisk's normal audio path
   - Requires special configuration to capture audio

2. **AUDIOHOOK_INHERIT is the solution**
   - Documented Asterisk feature for this exact scenario
   - Makes audio hooks follow channels through bridges

3. **Neither bot system records audio natively**
   - AVR: No recording environment variables or code
   - Pipecat: Only transcript recording (conversation text logs)
   - Asterisk-level recording is correct approach for both

4. **The `,b` option doesn't work with AudioSocket**
   - `,b` = record only audio TO the bot (one direction)
   - AudioSocket needs both directions for proper recording
   - Removing option allows full bidirectional capture

5. **Session summary was misleading**
   - Claimed Pipecat had "built-in recording capability"
   - Actually meant transcript recording, not audio WAV files
   - Led to confusion about which system to use
